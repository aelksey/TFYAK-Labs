## Теория для защиты лабораторной работы №5

### 1. Основная идея восходящего синтаксического анализа

При восходящем анализе входная цепочка терминалов **сворачивается** (редуцируется) в начальный нетерминал грамматики. Процесс идёт **снизу вверх** — от листьев дерева разбора к корню. На каждом шаге ищется **основа** (handle) — подцепочка, совпадающая с правой частью какого-либо правила, которая заменяется на нетерминал из левой части.

### 2. Структура автомата восходящего анализа

Восходящий анализатор (парсер) — это **конечный автомат со стековой памятью**, управляемый:
- текущим состоянием (номером, находящимся на вершине стека);
- текущим входным символом.

**Стек** хранит номера состояний. Перед запуском в стек помещается начальное состояние (0).

### 3. Операции восходящего анализатора

| Операция | Обозначение | Действие |
|----------|-------------|----------|
| **Shift** (сдвиг) | S*n* | Прочитать следующий терминал, занести состояние *n* в стек, перейти в состояние *n* |
| **Go** (переход) | G*n* | Занести состояние *n* в стек, перейти в состояние *n* (текущий входной символ не меняется) |
| **Reduce** (свёртка) | R*k*,*n* | Сбросить *k* состояний с вершины стека (по длине правой части), затем выполнить Go по нетерминалу *n* |
| **Stop** | Stop | Останов по успешному завершению разбора (в пакете — S-1) |

### 4. Конфигурации

**Конфигурация** — это правило грамматики с маркером (▼), показывающим текущую позицию разбора:

> *N* : α ▼ β

- α — уже обработанная часть
- β — ещё не обработанная
- Маркер может находиться в любом месте правой части (включая начало и конец)

Для правила длины *k* существует *k+1* конфигурация.

### 5. Ожидаемый правый контекст (lookahead)

**Ожидаемый правый контекст (ОПК)** — множество терминалов (включая ►), которые могут следовать за данной конфигурацией в правильно построенном дереве разбора.

Использование ОПК позволяет разрешать конфликты «сдвиг/свёртка» и «свёртка/свёртка».

### 6. Классы LR-грамматик

| Класс | Конфигурации | Разрешение конфликтов | Применение |
|-------|--------------|----------------------|------------|
| **LR(0)** | Простые (без ОПК) | Без использования lookahead | Теоретический интерес, узкий класс |
| **SLR(1)** | Простые | ОПК = множества последователей нетерминалов | Простые языки |
| **LALR(1)** | Расширенные (ОПК — множества) | Объединение канонических конфигураций | **Практически все языки программирования** |
| **LR(1)** | Канонические (ОПК — один символ) | Максимальная мощность, много состояний | Теоретическая основа |

### 7. Алгоритм построения таблицы расширенных конфигураций (LALR(1))

**Шаг 1. Начальное состояние**  
Состояние 0: `Z : ▼ S ► , { ► }`

**Шаг 2. Замыкание (closure)**  
Если есть конфигурация `N : α ▼ X β , {t₀}`, то для каждого правила `X → γ` добавить `X : ▼ γ` с ОПК, вычисляемым по правилу:
- если β содержит терминал или неаннулируемый нетерминал → ОПК = *M*ₚᵣₑ(β)
- если β = ε → ОПК = {t₀}
- если β из аннулируемых нетерминалов → ОПК = *M*ₚᵣₑ(β) ∪ {t₀}

При совпадении маркированных правил ОПК объединяются.

**Шаг 3. Переход (goto)**  
Для каждого символа *X*, перед которым стоит ▼, перенести маркер через *X* → новая база. ОПК сохраняется.

**Шаг 4. Слияние состояний**  
Если новое состояние имеет тот же набор маркированных правил, что и существующее, они сливаются (ОПК объединяются). ОПК дочерних конфигураций пересчитываются рекурсивно.

**Шаг 5. Построение управляющей таблицы**  
- **Shift** — если маркер перед терминалом
- **Go** — если маркер перед нетерминалом
- **Reduce** — если маркер в конце правила, **только** в столбцы, символы которых входят в ОПК
- **Stop** — в состоянии, соответствующем завершению

### 8. Преобразование таблицы конфигураций в управляющую таблицу

1. Строки — состояния (0..N)
2. Столбцы — терминалы, нетерминалы, ►
3. Заполнение по правилам из шага 5
4. Пустые клетки — ошибка

### 9. Разрешение конфликтов

| Конфликт | Причина | Разрешение в SLR(1)/LALR(1) |
|----------|---------|----------------------------|
| **Сдвиг/свёртка** | В одной клетке возможны Shift и Reduce | Reduce заносится только в столбцы, входящие в ОПК |
| **Свёртка/свёртка** | В одной клетке возможны два Reduce | Если ОПК правил не пересекаются — конфликта нет |

### 10. Пример работы (на грамматике арифметических выражений)

Для грамматики *G~a~₁* (с левой рекурсией) после построения конфигураций и использования ОПК управляющая таблица не содержит конфликтов. Состояния: 0…12, операции S, G, R.

### 11. Ключевые определения для защиты

- **Основа (handle)** — подцепочка, свёртываемая на данном шаге
- **Замыкание (closure)** — добавление всех возможных конфигураций, начинающихся с нетерминала после маркера
- **База состояния** — множество конфигураций, полученных переносом маркера
- **Каноническая конфигурация** — с одним символом ОПК
- **Расширенная конфигурация** — с множеством символов ОПК

### 12. Сравнение с нисходящим анализом

| Характеристика | Нисходящий (LL) | Восходящий (LR) |
|----------------|-----------------|-----------------|
| Направление | От корня к листьям | От листьев к корню |
| Операции | Развёртывание нетерминалов | Свёртка правых частей |
| Использование стека | Для возвратов / состояний | Для хранения состояний |
| Класс грамматик | LL(1), LL(k) | LR(0), SLR(1), LALR(1), LR(1) |
| Мощность | Меньше | **Больше** (распознаёт больше языков) |
