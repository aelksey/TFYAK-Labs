## 6.1) Что такое множество выбора правила грамматики?

**Ответ:**

**Множество выбора (SELECT)** правила `A → α` — это множество терминалов, при появлении которых в качестве текущего символа входной строки парсер должен применить именно это правило.

**Формальное определение:**
```
SELECT(A → α) = FIRST(α) ∪ (FOLLOW(A), если ε ∈ FIRST(α))
```

**Правила вычисления:**

1. Если `ε ∉ FIRST(α)`:
   ```
   SELECT(A → α) = FIRST(α)
   ```

2. Если `ε ∈ FIRST(α)`:
   ```
   SELECT(A → α) = (FIRST(α) \ {ε}) ∪ FOLLOW(A)
   ```

**Свойство LL(1):** Грамматика является LL(1) тогда и только тогда, когда для каждого нетерминала A все его правила имеют непересекающиеся множества SELECT.

```
SELECT(A → α₁) ∩ SELECT(A → α₂) ∩ ... ∩ SELECT(A → αₙ) = ∅
```

**Пример:**
```
Правила:          E' → +TE' | ε
FIRST(+TE') = {+}
FIRST(ε) = {ε}
FOLLOW(E') = {$, )}

SELECT(E' → +TE') = {+}
SELECT(E' → ε) = FOLLOW(E') = {$, )}

Пересечение: {+} ∩ {$, )} = ∅ → LL(1)
```

## 6.3) Напишите LL(1)-грамматику для условного оператора С-подобного языка, имеющего и полную и сокращенную форму.

**Ответ:**

**Проблема:** Классический "dangling else" делает грамматику не LL(1).

**LL(1) решение:**

```
statement      → if_statement | other_statement
if_statement   → "if" "(" condition ")" statement else_part
else_part      → "else" statement | ε
condition      → ... (выражение)
other_statement → ... (другие операторы)
```

**Однако** это не решает проблему полностью. Лучшее решение — **разделить на два нетерминала**:

```
statement        → matched_statement | unmatched_statement
matched_statement → "if" "(" condition ")" matched_statement "else" matched_statement
                  | other_statement
unmatched_statement → "if" "(" condition ")" statement
                  | "if" "(" condition ")" matched_statement "else" unmatched_statement
```

**Полная LL(1) грамматика:**
```
program          → statement*
statement        → matched | unmatched
matched          → "if" "(" expr ")" matched "else" matched
                 | other
unmatched        → "if" "(" expr ")" statement
                 | "if" "(" expr ")" matched "else" unmatched
other            → id "=" expr ";" | "while" "(" expr ")" statement | ...
expr             → ... (выражение)
```

**Пример разбора:**
```c
// Сокращённая форма
if (x > 0) y = 1;                    // unmatched

// Полная форма  
if (x > 0) y = 1; else y = 2;        // matched

// Вложенный с dangling else
if (a > 0) if (b > 0) x = 1; else x = 2;  // else привязан к внутреннему if
```

## 6.4) Что такое s-грамматика?

**Ответ:**

**S-грамматика (Simple grammar)** — это простейший подкласс LL(1) грамматик, в котором:

1. **Нет ε-правил** (все правила порождают хотя бы один терминал)
2. **Каждое правило начинается с терминала**
3. **Все правые части уникальны по первому символу**

**Формальное определение:**

Для каждого нетерминала A:
- Все правила имеют вид `A → aα`, где `a` — терминал
- Для разных правил `A → aα` и `A → bβ` выполняется `a ≠ b`

**Пример S-грамматики:**
```
S → aA | bB
A → cA | d
B → eB | f
```

**Не S-грамматика:**
```
S → Aa        // начинается с нетерминала
A → ε         // есть ε-правило
A → aB | aC   // одинаковые первые символы
```

**Свойства:**
- Всегда LL(1)
- Простейший для реализации
- Не требует вычисления FOLLOW
- Ограниченная выразительная мощность

## 6.5) Какие операции способен выполнять автомат нисходящего восстановления дерева грамматического разбора с одним состоянием?

**Ответ:**

Автомат нисходящего разбора с одним состоянием выполняет следующие операции:

### 1. **Разворачивание (Expand)**
- Замена нетерминала на правую часть правила
- Чтение из таблицы: `TABLE[A, a] = A → α`
- Помещение α в стек (в обратном порядке)

### 2. **Свертка/Сопоставление (Match)**
- Сравнение верхушки стека (терминала) с текущим символом ввода
- При совпадении: снять вершину стека, продвинуть указатель ввода
- При несовпадении: синтаксическая ошибка

### 3. **Успешное завершение (Accept)**
- Стек содержит только `$` (маркер дна)
- Вся входная строка прочитана
- Переход в финальное состояние

### 4. **Обработка ошибки (Error)**
- В таблице нет записи для пары `(A, a)`
- Вызов процедуры восстановления после ошибки

**Схема работы:**
```
┌─────────────────────────────────────────┐
│  Входная строка: a₁ a₂ ... aₙ $         │
├─────────────────────────────────────────┤
│  Стек:          $ S                     │
├─────────────────────────────────────────┤
│  Управляющая таблица: M[A, a]           │
└─────────────────────────────────────────┘

Операции:
1. Снять X с вершины стека
2. Если X - терминал → MATCH
3. Если X - нетерминал → EXPAND
4. Если стек пуст и ввод = $ → ACCEPT
```

## 6.6) Что такое LL(1)-грамматики?

**Ответ:**

**LL(1)-грамматика** — это грамматика, для которой существует детерминированный нисходящий парсер, использующий один символ предпросмотра.

**Расшифровка:**
- **L** — чтение слева направо (Left-to-right)
- **L** — левый вывод (Leftmost derivation)
- **1** — один символ предпросмотра

**Условия LL(1):**

1. **Нет левой рекурсии** (прямой или косвенной)

2. **Грамматика факторизована** (нет правил с общим префиксом)

3. **Для каждого нетерминала A с правилами A → α₁ | α₂ | ... | αₙ:**
   ```
   FIRST(αᵢ) ∩ FIRST(αⱼ) = ∅  для всех i ≠ j
   ```

4. **Для каждого правила A → ε:**
   ```
   FIRST(A) ∩ FOLLOW(A) = ∅
   ```

**Проверка через SELECT:**
```
SELECT(A → αᵢ) ∩ SELECT(A → αⱼ) = ∅  для всех i ≠ j
```

**Пример LL(1) грамматики:**
```
E  → TE'
E' → +TE' | ε
T  → FT'
T' → *FT' | ε
F  → (E) | id
```

**Пример НЕ LL(1) грамматики:**
```
S → if E then S else S
S → if E then S           // конфликт FIRST
A → Aa | b                // левая рекурсия
```

## 6.7) Что такое детерминированное (безвозвратное) восстановление дерева грамматического разбора?

**Ответ:**

**Детерминированное (безвозвратное) восстановление дерева разбора** — это процесс построения дерева разбора, при котором на каждом шаге принимается единственное решение, не требующее возврата (backtracking) к предыдущим состояниям.

### Ключевые характеристики:

1. **Отсутствие backtracking** — решение о применении правила принимается окончательно

2. **Один символ предпросмотра** — достаточно для выбора правила

3. **Линейное время** — O(n), где n — длина строки

4. **Детерминированность** — для каждой пары (нетерминал, текущий символ) существует не более одного правила

### Сравнение:

| Характеристика | Детерминированный | Недетерминированный (с возвратами) |
|----------------|-------------------|-----------------------------------|
| Время работы | O(n) | O(2ⁿ) в худшем случае |
| Память | O(n) | O(n²) |
| Сложность реализации | Низкая | Высокая |
| Предпросмотр | 1 символ | Неограниченный |

### Пример:

**Детерминированный разбор (LL(1)):**
```
Ввод: (x)
Стек: $S

Шаг 1: видя '(', применяем S→(S) → без вариантов
Шаг 2: видя 'x', применяем S→x → без вариантов
→ Успех без возвратов
```

**Недетерминированный разбор (с возвратами):**
```
S → (S) | (S)S | x

Шаг 1: видя '(', есть 2 варианта:
  - пробуем S→(S) → неудача
  - возвращаемся, пробуем S→(S)S → успех
```

## 6.8) Почему леворекурсивная грамматика не может быть преобразована в нисходящий синтаксический акцептор?

**Ответ:**

Леворекурсивная грамматика приводит к **бесконечному циклу** в нисходящем парсере.

### Причина:

Для правила вида `A → Aα` парсер:

1. Начинает раскрывать A
2. Видит, что нужно раскрыть A снова
3. Снова пытается раскрыть A
4. И так бесконечно

**Демонстрация:**

```
Грамматика: E → E + T | T

Процесс разбора при виде 'id':
1. Стек: $E, ввод: id...
2. Нужно раскрыть E → E + T (первое правило)
3. Помещаем: + T E в стек (в обратном порядке)
4. Стек: $+ T E, ввод: id...
5. На вершине E → снова раскрываем E → E + T
6. Помещаем: + T E в стек
7. Стек: $+ T + T E → БЕСКОНЕЧНО
```

### Почему нельзя просто "взять и раскрыть"?

Нисходящий партер **не может знать**, сколько раз нужно применить леворекурсивное правило, потому что:
- Он не видит всю строку целиком
- Он не может "заглянуть вперёд" на произвольное количество символов

### Решение:

Левая рекурсия **устраняется** преобразованием грамматики:

```
Было:  E → E + T | T
Стало: E → T E'
       E' → + T E' | ε
```

После преобразования рекурсия становится **правой**, что допустимо для нисходящего разбора.

## 6.9) Опишите технологию разработки процедурной реализации рекурсивного спуска.

**Ответ:**

**Рекурсивный спуск** — метод реализации нисходящего парсера, где для каждого нетерминала пишется отдельная процедура.

### Технология разработки:

#### Шаг 1: Подготовка грамматики
- Устранить левую рекурсию
- Провести левую факторизацию
- Привести к LL(1) виду

#### Шаг 2: Определение структуры
```c
// Глобальные переменные
char* input;        // входная строка
int pos;            // текущая позиция
int error;          // флаг ошибки
```

#### Шаг 3: Вспомогательные функции
```c
char current() { return input[pos]; }
void next() { pos++; }
void match(char expected) {
    if (current() == expected) next();
    else error = 1;
}
```

#### Шаг 4: Процедуры для нетерминалов
```c
void E() {   // E → T E'
    T();
    Eprime();
}

void Eprime() {   // E' → + T E' | ε
    if (current() == '+') {
        match('+');
        T();
        Eprime();
    }
    // иначе ε
}

void T() {   // T → F T'
    F();
    Tprime();
}

void Tprime() {   // T' → * F T' | ε
    if (current() == '*') {
        match('*');
        F();
        Tprime();
    }
}

void F() {   // F → (E) | id
    if (current() == '(') {
        match('(');
        E();
        match(')');
    } else if (isalpha(current())) {
        match(current());  // id
    } else {
        error = 1;
    }
}
```

#### Шаг 5: Главная функция
```c
int parse(char* str) {
    input = str;
    pos = 0;
    error = 0;
    E();
    if (!error && current() == '\0') return 1;
    return 0;
}
```

### Преимущества процедурной реализации:
- Простота отладки
- Легко добавить семантические действия
- Естественное отображение грамматики на код

## 6.10) Как вычисляются множества выбора правил грамматики?

**Ответ:**

### Алгоритм вычисления SELECT(A → α)

**Шаг 1:** Вычислить FIRST(α)

**Шаг 2:** Если ε ∉ FIRST(α):
```
SELECT(A → α) = FIRST(α)
```

**Шаг 3:** Если ε ∈ FIRST(α):
```
SELECT(A → α) = (FIRST(α) \ {ε}) ∪ FOLLOW(A)
```

### Пример вычисления

**Грамматика:**
```
1. E  → T E'
2. E' → + T E'
3. E' → ε
4. T  → F T'
5. T' → * F T'
6. T' → ε
7. F  → (E)
8. F  → id
```

**FIRST и FOLLOW:**
```
FIRST(E)   = FIRST(T) = {(, id}
FIRST(E')  = {+, ε}
FIRST(T)   = {(, id}
FIRST(T')  = {*, ε}
FIRST(F)   = {(, id}

FOLLOW(E)  = {$, )}
FOLLOW(E') = {$, )}
FOLLOW(T)  = {+, $, )}
FOLLOW(T') = {+, $, )}
FOLLOW(F)  = {*, +, $, )}
```

**Вычисление SELECT:**

```
SELECT(E  → T E')   = FIRST(T E') = FIRST(T) = {(, id}
SELECT(E' → + T E') = {+}
SELECT(E' → ε)      = FOLLOW(E') = {$, )}
SELECT(T  → F T')   = FIRST(F) = {(, id}
SELECT(T' → * F T') = {*}
SELECT(T' → ε)      = FOLLOW(T') = {+, $, )}
SELECT(F  → (E))    = {(}
SELECT(F  → id)     = {id}
```

### Проверка LL(1):

| Нетерминал | SELECT множества | Пересечение |
|------------|------------------|-------------|
| E | {(, id} | ∅ → OK |
| E' | {+} и {$, )} | ∅ → OK |
| T | {(, id} | ∅ → OK |
| T' | {*} и {+, $, )} | ∅ → OK |
| F | {(} и {id} | ∅ → OK |

## 6.11) Перечислите поля клетки нисходящего автомата с несколькими состояниями.

**Ответ:**

В нисходящем автомате с несколькими состояниями каждая клетка управляющей таблицы содержит следующие поля:

### Поля клетки M[состояние, входной_символ]:

| Поле | Обозначение | Назначение |
|------|-------------|------------|
| **Флаг действия** | `act` | Тип операции: развернуть, свернуть, принять, ошибка |
| **Номер правила** | `rule_num` | Номер грамматического правила для разворачивания |
| **Следующее состояние** | `next_state` | Состояние, в которое переходит автомат |
| **Флаг ошибки** | `err_flag` | Индикатор синтаксической ошибки |
| **Счётчик символов** | `count` | Количество символов для свертки (для LR-подобных автоматов) |

### Структура клетки (в виде записи):
```
struct TableCell {
    int action;        // 0=error, 1=expand, 2=match, 3=accept
    int rule;          // номер правила (для expand)
    int next_state;    // следующее состояние автомата
    int error_code;    // код ошибки
};
```

### Пример заполнения:

| Состояние | Вход | action | rule | next_state |
|-----------|------|--------|------|------------|
| 0 | id | expand | F→id | 1 |
| 0 | ( | expand | F→(E) | 2 |
| 1 | + | match | - | 3 |
| 2 | id | expand | ... | ... |

## 6.12) Как формируются значения флагов в управляющей таблице нисходящего синтаксического акцептора с несколькими состояниями?

**Ответ:**

### Процесс формирования флагов:

#### Шаг 1: Построение FIRST и FOLLOW множеств
Вычисляются для всех нетерминалов и правил.

#### Шаг 2: Вычисление SELECT множеств
```
SELECT(A → α) = FIRST(α) ∪ (FOLLOW(A) если ε ∈ FIRST(α))
```

#### Шаг 3: Формирование таблицы

Для каждого правила `A → α` (с номером i):

1. **Для каждого терминала a ∈ SELECT(A → α):**
   - `M[state_A, a].action = EXPAND`
   - `M[state_A, a].rule = i`

2. **Для каждого терминала a ∈ FOLLOW(A) (если есть ε-правило):**
   - `M[state_A, a].action = REDUCE` (или EXPAND с ε-правилом)

#### Шаг 4: Заполнение терминальных переходов

Для каждого терминала a:
- `M[state_a, a].action = MATCH`
- `M[state_a, a].next_state = следующее_состояние`

#### Шаг 5: Заполнение ошибок

Все незаполненные клетки:
- `M[state, symbol].action = ERROR`

### Пример формирования:

**Грамматика:** E → T E', E' → + T E' | ε

| Состояние | Символ | action | rule |
|-----------|--------|--------|------|
| state_E | id | EXPAND | 1 |
| state_E | ( | EXPAND | 1 |
| state_E' | + | EXPAND | 2 |
| state_E' | ) | EXPAND | 3 (ε) |
| state_E' | $ | EXPAND | 3 (ε) |
| state_T | id | EXPAND | 4 |
| state_T | ( | EXPAND | 4 |

## 6.13) Разработайте LL(1)-грамматику для оператора переключателя (switch) С-подобного языка.

**Ответ:**

### Проблемы switch в LL(1):
- Несколько вариантов `case`
- Опциональный `default`
- Возможен `break`

### LL(1) грамматика для switch:

```
statement          → switch_statement | other_statement
switch_statement   → "switch" "(" expr ")" "{" case_list default_part "}"
case_list          → case_case case_list | ε
case_case          → "case" const ":" statement_list
default_part       → "default" ":" statement_list | ε
statement_list     → statement statement_list | ε
other_statement    → ... (другие операторы)
break_statement    → "break" ";"
```

### Полная LL(1) грамматика:

```
program            → statement* EOF

statement          → switch_stmt | other_stmt | break_stmt
switch_stmt        → "switch" "(" expr ")" "{" cases default "}"
cases              → case cases | ε
case               → "case" const ":" stmt_list
default            → "default" ":" stmt_list | ε
stmt_list          → statement stmt_list | ε
break_stmt         → "break" ";"

other_stmt         → assignment | condition | cycle | returnValue
expr               → ... (выражение)
const              → ... (константа)
```

### Пример корректной конструкции:
```c
switch (x) {
    case 1:
        a = 10;
        break;
    case 2:
        b = 20;
        break;
    default:
        c = 30;
}
```

### Проверка LL(1):

| Нетерминал | SELECT множества |
|------------|------------------|
| cases | FIRST(case) = {"case"} и ε (FOLLOW = {"default", "}"}) → непересекаются |
| default | FIRST("default") = {"default"} и ε (FOLLOW = {"}"}) → непересекаются |
| stmt_list | FIRST(statement) и ε (FOLLOW = {"case", "default", "}"}) → непересекаются |

## 6.14) В каком порядке нумеруются символы грамматики при преобразовании в управляющую таблицу автомата с несколькими состояниями?

## 6.15) Чем управляется нисходящий автомат с одним состоянием?

**Ответ:**

Нисходящий автомат с одним состоянием управляется **управляющей таблицей M[A, a]**, где:
- `A` — нетерминал (текущая вершина стека)
- `a` — текущий символ ввода

### Элементы управления:

1. **Стек** — хранит ожидаемые символы (терминалы и нетерминалы)

2. **Входная лента** — содержит разбираемую строку с маркером конца `$`

3. **Управляющая таблица** — определяет действие для каждой пары `(нетерминал, символ)`

4. **Счётчик позиции** — указатель текущего символа ввода

### Формат управляющей таблицы:

```
M[нетерминал, терминал] = правило | error
```

### Процесс управления:

```
┌──────────────────────────────────────────────────┐
│  Управляющая таблица M[A, a]                      │
│  ┌─────┬───┬───┬───┬───┬───┬─────┐               │
│  │     │ + │ * │ ( │ ) │id │  $  │               │
│  ├─────┼───┼───┼───┼───┼───┼─────┤               │
│  │  E  │   │   │TE'│   │TE'│     │               │
│  │  E' │+TE'│   │   │ ε │   │  ε  │               │
│  │  T  │   │   │FT'│   │FT'│     │               │
│  │  T' │ ε │*FT'│   │ ε │   │  ε  │               │
│  │  F  │   │   │(E)│   │id │     │               │
│  └─────┴───┴───┴───┴───┴───┴─────┘               │
└──────────────────────────────────────────────────┘
         ↓
┌──────────────────────────────────────────────────┐
│  Стек: $ S                                        │
│  Ввод: a₁ a₂ ... aₙ $                            │
└──────────────────────────────────────────────────┘
```

## 6.16) Каково назначение каждого управляющего флажка управляющей таблицы автомата с несколькими состояниями?

**Ответ:**

В автомате с несколькими состояниями каждая клетка таблицы содержит несколько управляющих флажков (полей):

### Поля и их назначение:

| Флажок | Назначение |
|--------|------------|
| **EXPAND** | Развернуть нетерминал по правилу grammar[rule] |
| **MATCH** | Сопоставить терминал с входным символом |
| **ACCEPT** | Успешное завершение разбора |
| **ERROR** | Синтаксическая ошибка |
| **POP** | Удалить вершину стека |
| **PUSH** | Поместить символы на стек |
| **SHIFT** | Сдвиг (продвижение по входу) |
| **REDUCE** | Свертка (замена цепочки нетерминалом) |
| **GOTO** | Переход в новое состояние |

### Подробное описание:

1. **EXPAND (E)**
   - Применяется к нетерминалу на вершине стека
   - Заменяет его правой частью правила
   - Аргумент: номер правила

2. **MATCH (M)**
   - Применяется к терминалу на вершине стека
   - Сравнивает с текущим символом ввода
   - При совпадении: снимает вершину, сдвигает ввод

3. **ACCEPT (A)**
   - Разбор завершён успешно
   - Стек пуст, ввод прочитан до конца

4. **ERROR (E)**
   - Ошибочная ситуация
   - Нет правила для текущей пары (состояние, символ)

5. **POP (P)**
   - Удаляет символ с вершины стека
   - Используется при обработке ε-правил

6. **PUSH (U)**
   - Помещает последовательность символов на стек
   - Обычно используется с EXPAND

7. **SHIFT (S)**
   - Продвигает указатель ввода на следующий символ
   - Используется в LR-подобных автоматах

8. **REDUCE (R)**
   - Заменяет цепочку символов на нетерминал
   - Используется в восходящих парсерах

9. **GOTO (G)**
   - Переход в новое состояние автомата
   - Используется в автоматах с несколькими состояниями

## 6.17) Как формируются значения управляющих флагов нисходящего автомата с несколькими состояниями?

**Ответ:**

### Алгоритм формирования:

#### Шаг 1: Построение множеств SELECT
Для каждого правила `A → α` вычисляется `SELECT(A → α)`

#### Шаг 2: Построение состояний
Каждый нетерминал и терминал получает своё состояние

#### Шаг 3: Заполнение флагов для нетерминалов

Для каждого нетерминала A и каждого символа a:
```
if a ∈ SELECT(A → α):
    M[A, a].action = EXPAND
    M[A, a].rule = номер_правила
```

#### Шаг 4: Заполнение флагов для терминалов

Для каждого терминала a:
```
M[a, a].action = MATCH
M[a, a].next_state = next_state
```

#### Шаг 5: Заполнение флагов для маркера конца

```
M[$, $].action = ACCEPT
```

#### Шаг 6: Заполнение остальных клеток

```
M[state, symbol].action = ERROR
```

### Пример формирования:

**Грамматика:** E → T E', E' → + T E' | ε

**SELECT множества:**
```
SELECT(E → T E')   = {(, id}
SELECT(E' → + T E') = {+}
SELECT(E' → ε)      = {$, )}
```

**Таблица с флагами:**

| Состояние | Вход | action | rule | next_state |
|-----------|------|--------|------|------------|
| E | ( | EXPAND | 1 | - |
| E | id | EXPAND | 1 | - |
| E | + | ERROR | - | - |
| E | ) | ERROR | - | - |
| E | $ | ERROR | - | - |
| E' | + | EXPAND | 2 | - |
| E' | ) | EXPAND | 3 | - |
| E' | $ | EXPAND | 3 | - |
| T | ( | EXPAND | 4 | - |
| T | id | EXPAND | 4 | - |
| ( | ( | MATCH | - | T |
| ) | ) | MATCH | - | E' |
| id | id | MATCH | - | T |
| $ | $ | ACCEPT | - | - |

## 6.18) Опишите последовательность действий при преобразовании грамматики в процедурную реализацию нисходящего синтаксического акцептора.

**Ответ:**

### Последовательность действий:

#### Этап 1: Подготовка грамматики
```
1. Устранить левую рекурсию
2. Провести левую факторизацию
3. Привести грамматику к LL(1) виду
4. Проверить выполнение условий LL(1)
```

#### Этап 2: Вычисление множеств
```
5. Вычислить FIRST для всех символов
6. Вычислить FOLLOW для всех нетерминалов
7. Вычислить SELECT для каждого правила
```

#### Этап 3: Проектирование процедур
```
8. Определить набор процедур (по числу нетерминалов)
9. Определить глобальные переменные (вход, позиция, ошибка)
10. Написать вспомогательные функции (match, current, next)
```

#### Этап 4: Реализация процедур
```
11. Для каждого нетерминала A:
    - Посмотреть SELECT(A → α)
    - Для каждой альтернативы:
        * Проверить lookahead символ
        * Вызвать соответствующие процедуры/матчить терминалы
12. Добавить обработку ошибок
```

#### Этап 5: Интеграция
```
13. Написать главную функцию parse()
14. Добавить инициализацию
15. Вызвать стартовую процедуру
16. Проверить успешность разбора
```

#### Этап 6: Тестирование
```
17. Протестировать на корректных строках
18. Протестировать на некорректных строках
19. Отладить обработку ошибок
```

### Пример реализации:

```c
// Этап 1-2: Грамматика подготовлена
// E → T E'
// E' → + T E' | ε
// T → F T'
// T' → * F T' | ε
// F → (E) | id

// Этап 3: Глобальные переменные
char* input;
int pos;
int error;

// Этап 4: Реализация
void E() {
    // SELECT(E → T E') = {(, id}
    if (current() == '(' || isalpha(current())) {
        T();
        Eprime();
    } else {
        error = 1;
    }
}

void Eprime() {
    // SELECT(E' → + T E') = {+}
    if (current() == '+') {
        match('+');
        T();
        Eprime();
    }
    // SELECT(E' → ε) = {$, )}
    // иначе ничего не делаем (ε)
}

void T() {
    if (current() == '(' || isalpha(current())) {
        F();
        Tprime();
    } else {
        error = 1;
    }
}

void Tprime() {
    if (current() == '*') {
        match('*');
        F();
        Tprime();
    }
}

void F() {
    if (current() == '(') {
        match('(');
        E();
        match(')');
    } else if (isalpha(current())) {
        match(current());
    } else {
        error = 1;
    }
}

// Этап 5: Главная функция
int parse(char* str) {
    input = str;
    pos = 0;
    error = 0;
    E();
    if (!error && current() == '\0') return 1;
    return 0;
}
```

## 6.19) Как формируются множества выбора состояний нисходящего синтаксического акцептора с несколькими состояниями?

**Ответ:**

### Определение

**Множество выбора состояния (SELECT_state)** — множество терминалов, при появлении которых автомат переходит в данное состояние.

### Алгоритм формирования:

#### Шаг 1: Определение состояний
Каждый нетерминал и терминал получает своё состояние.

#### Шаг 2: Для каждого состояния (нетерминала)
```
SELECT_state(A) = { a | M[A, a] ≠ error }
```
То есть все терминалы, для которых в таблице определено действие.

#### Шаг 3: Для каждого состояния (терминала)
```
SELECT_state(a) = {a}
```
Терминал может быть сопоставлен только с самим собой.

#### Шаг 4: Формирование по правилам

Для каждого правила `A → α`:
```
SELECT_rule(A → α) = SELECT(A → α)
```
Множество выбора правила равно SELECT этого правила.

### Пример:

**Грамматика:** E → T E', E' → + T E' | ε, T → F T', T' → * F T' | ε, F → (E) | id

**SELECT_state:**

| Состояние | SELECT_state |
|-----------|--------------|
| E | {(, id} |
| E' | {+, $, )} |
| T | {(, id} |
| T' | {*, +, $, )} |
| F | {(, id} |
| ( | {(} |
| ) | {)} |
| + | {+} |
| * | {*} |
| id | {id} |
| $ | {$} |

### Проверка непротиворечивости:

Для каждого состояния (нетерминала) объединение SELECT_state всех его правил должно равняться SELECT_state самого состояния.

```
SELECT_state(E) = SELECT(E → T E') = {(, id)} ✓
SELECT_state(E') = SELECT(E' → + T E') ∪ SELECT(E' → ε) = {+} ∪ {$, )} = {+, $, )} ✓
```

## 6.20) Опишите процесс функционирования нисходящего рекурсивного спуска.

**Ответ:**

### Процесс функционирования:

#### Инициализация
```
1. Установить указатель ввода на начало строки
2. Обнулить флаг ошибки
3. Вызвать процедуру для стартового нетерминала
```

#### Основной цикл (в процедурах)
```
Для каждой процедуры нетерминала A:

1. Определить текущий символ ввода (lookahead)
2. Найти правило A → α, где lookahead ∈ SELECT(A → α)
3. Выполнить правую часть правила:
   - Для каждого символа β в α (слева направо):
     * Если β - терминал:
       - Сравнить с текущим символом
       - При совпадении: прочитать символ, продвинуть указатель
       - При несовпадении: сигнал ошибки
     * Если β - нетерминал:
       - Рекурсивно вызвать процедуру для β
4. Вернуться в вызывающую процедуру
```

#### Обработка ε-правил
```
Если выбрано правило A → ε:
   - Ничего не читать
   - Сразу вернуться
```

#### Завершение разбора
```
1. После вызова стартовой процедуры
2. Проверить, что вся строка прочитана
3. Проверить, что нет ошибок
4. Вернуть успех/неудачу
```

### Пример работы (разбор "x+y"):

**Грамматика:** E → T E', E' → + T E' | ε, T → F T', T' → * F T' | ε, F → id

**Процесс:**

```
Вызов E():
  current() = 'x' (id)
  Правило: E → T E'
  Вызов T():
    current() = 'x' (id)
    Правило: T → F T'
    Вызов F():
      current() = 'x' (id)
      Правило: F → id
      match('x') → успех, pos++ (теперь '+')
      return
    Вызов T'():
      current() = '+'
      Правило: T' → ε (так как нет '*')
      return
    return
  Вызов E'():
    current() = '+'
    Правило: E' → + T E'
    match('+') → успех, pos++ (теперь 'y')
    Вызов T():
      current() = 'y' (id)
      Правило: T → F T'
      Вызов F():
        current() = 'y' (id)
        Правило: F → id
        match('y') → успех, pos++ (теперь '\0')
        return
      Вызов T'():
        current() = '\0'
        Правило: T' → ε
        return
      return
    Вызов E'():
      current() = '\0'
      Правило: E' → ε
      return
    return
  return

Проверка: pos в конце, error = 0 → УСПЕХ
```

### Схема вызовов:

```
E()
├── T()
│   ├── F()
│   │   └── match(id)
│   └── T'()
│       └── ε
└── E'()
    ├── match(+)
    ├── T()
    │   ├── F()
    │   │   └── match(id)
    │   └── T'()
    │       └── ε
    └── E'()
        └── ε
```

### Особенности рекурсивного спуска:

| Характеристика | Описание |
|----------------|----------|
| **Сложность** | O(n) по времени, O(h) по памяти (h — глубина рекурсии) |
| **Простота** | Высокая, код соответствует грамматике |
| **Обработка ошибок** | Лёгкая, можно на любой уровень |
| **Семантические действия** | Легко встраиваются |
