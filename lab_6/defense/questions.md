# Контрольные вопросы к лабораторной работе №6

## «Синтаксис языков программирования. Преобразование транслируемой программы в постфиксную форму записи»

Ниже приведены **развёрнутые ответы** на все контрольные вопросы из методического пособия.

---

### Вопрос 1. Для чего операторы программы преобразуются в постфиксную форму записи?

**Ответ:**

Операторы программы преобразуются в постфиксную форму записи (ПФЗ) для следующих целей:

1. **Упрощение генерации объектного кода** — ПФЗ позволяет легко преобразовывать выражения в машинные команды или байт-код виртуальной машины.

2. **Устранение неоднозначности** — в ПФЗ не требуется использовать скобки для указания порядка выполнения операций, так как сам порядок записи определяет последовательность вычислений.

3. **Совпадение порядка операций с порядком выполнения** — в ПФЗ знаки операций располагаются **после** своих операндов, что соответствует естественному порядку выполнения: сначала вычисляются операнды, затем выполняется операция.

4. **Упрощение интерпретации** — ПФЗ легко интерпретируется с использованием стекового механизма: операнды помещаются в стек, а при встрече оператора из стека извлекаются нужные операнды, результат помещается обратно.

5. **Универсальность** — ПФЗ является промежуточным представлением, не зависящим от конкретной архитектуры целевого процессора.

**Пример:** Выражение `a + b * c` в ПФЗ выглядит как `a b c * +`, что явно указывает: сначала умножить `b` на `c`, затем прибавить `a`.

---

### Вопрос 2. Эквивалентны ли абстрактное синтаксическое дерево и постфиксная форма записи программы?

**Ответ:**

**Нет, не полностью эквивалентны**, хотя и тесно связаны.

| Характеристика | Абстрактное синтаксическое дерево (АСД) | Постфиксная форма записи (ПФЗ) |
|----------------|------------------------------------------|--------------------------------|
| **Структура** | Иерархическая (дерево) | Линейная (последовательность) |
| **Наглядность** | Высокая (видна структура операций) | Низкая (трудно восстановить структуру) |
| **Использование памяти** | Больше (хранение связей между узлами) | Меньше (простой массив) |
| **Скорость обхода** | Медленнее (рекурсивный обход) | Быстрее (линейный проход) |
| **Информативность** | Сохраняет полную структуру программы | Сохраняет только порядок операций |

**Связь между ними:** ПФЗ может быть получена путём **обратного (постфиксного) обхода** АСД. При этом АСД содержит больше информации (например, типы узлов, вспомогательные данные), которая может быть утеряна при преобразовании в ПФЗ.

**Вывод:** ПФЗ является **линейным представлением** информации, содержащейся в АСД, но не эквивалентна ему, так как не сохраняет иерархическую структуру.

### Вопрос 3. Что такое детерминированное (безвозвратное) восстановление дерева грамматического разбора?

**Ответ:**

**Детерминированное (безвозвратное) восстановление дерева грамматического разбора** — это процесс синтаксического анализа, при котором на каждом шаге анализатор однозначно определяет, какое правило грамматики нужно применить, **без необходимости возврата (backtracking)** и перебора альтернатив.

**Основные характеристики:**

1. **Однозначность выбора** — для каждого текущего состояния и входного символа существует не более одного возможного действия (правила).

2. **Линейная сложность** — время анализа пропорционально длине входной цепочки: **O(n)**, где *n* — количество лексем.

3. **Отсутствие возвратов** — анализатор никогда не "откатывается" назад для пересмотра решения.

**Как достигается:**

- Для **нисходящего** анализа: грамматика должна быть **LL(k)**-грамматикой (обычно LL(1)).
- Для **восходящего** анализа: грамматика должна быть **LR(k)**-грамматикой (обычно LALR(1)).

**Преимущества:**
- Высокая скорость работы
- Простота реализации
- Предсказуемое использование памяти

**Пример недетерминированного (с возвратами) анализа:**
Грамматика: `S → aSa | aa`
Для цепочки `aaaa` анализатор может ошибочно выбрать `S → aSa` и затем выполнять возврат.

**Пример детерминированного анализа:**
Грамматика LL(1): `S → aB | bC; B → aS; C → bS` — по первому символу всегда ясно, какое правило применить.

### Вопрос 4. Сформулируйте основные принципы преобразования арифметических выражений в постфиксную запись.

**Ответ:**

Основные принципы преобразования инфиксных выражений в постфиксную форму записи (ПФЗ):

#### Принцип 1: Операнды сохраняют порядок
Операнды (идентификаторы, константы) в ПФЗ записываются в том же порядке, в котором они встречаются в исходном выражении.

#### Принцип 2: Операторы записываются после операндов
Для бинарной операции вида `A ○ B` в ПФЗ сначала записываются операнды `A` и `B`, а затем оператор `○`: `A B ○`

#### Принцип 3: Учёт приоритета операций
Операции с более высоким приоритетом выполняются раньше. В ПФЗ оператор с более высоким приоритетом появляется раньше, чем оператор с низким приоритетом.

#### Принцип 4: Учёт ассоциативности
- **Левоассоциативные** операции (+, -, *, /) обрабатываются слева направо
- **Правоассоциативные** операции (например, возведение в степень `^`) обрабатываются справа налево

#### Принцип 5: Скобки не нужны
Скобки в ПФЗ не сохраняются, так как порядок операций определяется положением операторов.

#### Алгоритм (метод сортировочной станции Дейкстры):

```
Вход: инфиксное выражение
Выход: ПФЗ

Для каждого токена t в выражении слева направо:
    Если t — операнд:
        Добавить t в выходную строку
    Если t — открывающая скобка '(':
        Поместить t в стек операторов
    Если t — закрывающая скобка ')':
        Выталкивать операторы из стека в выходную строку, 
        пока не встретится '(' (которую затем удалить)
    Если t — оператор:
        Пока стек не пуст и на вершине стека оператор с приоритетом >= приоритета t
        и операторы левоассоциативные:
            Вытолкнуть оператор из стека в выходную строку
        Поместить t в стек
Конец цикла
Вытолкнуть все оставшиеся операторы из стека в выходную строку
```

#### Пример преобразования:

| Инфиксная запись | Постфиксная запись |
|------------------|---------------------|
| `a + b` | `a b +` |
| `a + b * c` | `a b c * +` |
| `(a + b) * c` | `a b + c *` |
| `a - b - c` | `a b - c -` |
| `a ^ b ^ c` (правоассоциативная) | `a b c ^ ^` |

### Вопрос 5. Почему леворекурсивная грамматика не может быть преобразована в нисходящий синтаксический акцептор?

**Ответ:**

**Леворекурсивная грамматика** — это грамматика, содержащая правила вида `A → Aα` (прямая левая рекурсия) или цепочку правил, приводящую к такому же эффекту (косвенная левая рекурсия).

**Причина невозможности использования для нисходящего анализа:**

1. **Бесконечный цикл рекурсии**
   
   При попытке разбора с использованием метода рекурсивного спуска анализатор войдёт в бесконечную рекурсию:
   
   ```
   function parseA() {
       parseA();  // бесконечный вызов!
       // ... обработка α
   }
   ```

2. **Проблема выбора правила в LL(1)-анализаторах**
   
   Для правила `A → Aα | β` множества выбора (FIRST) пересекаются:
   - FIRST(Aα) = FIRST(A) ∪ FIRST(α)
   - FIRST(β) — также входит в FIRST(A)
   
   Это нарушает главное требование LL(1)-грамматик: **множества выбора правил с одинаковой левой частью должны быть попарно непересекающимися**.

3. **Невозможность определить правило по первому символу**
   
   В LL(1)-анализаторе решение о выборе правила принимается на основе текущего входного символа (lookahead). При левой рекурсии этот символ одинаков для всех рекурсивных правил.

**Пример леворекурсивной грамматики:**
```
E → E + T | T
T → T * F | F
F → id | (E)
```

**Как исправить:** Устранить левую рекурсию, преобразовав грамматику:

```
E → T E'
E' → + T E' | ε
T → F T'
T' → * F T' | ε
F → id | (E)
```

**Итог:** Нисходящие анализаторы (LL-семейство) требуют, чтобы грамматика была **нелеворекурсивной**. Леворекурсивные грамматики могут анализироваться только **восходящими** методами (LR-семейство).

### Вопрос 6. Что такое постфиксная запись?

**Ответ:**

**Постфиксная запись** (также известна как **обратная польская запись**, RPN — Reverse Polish Notation) — это форма записи математических и логических выражений, в которой **знаки операций располагаются после своих операндов**.

---

#### Историческая справка

Разработана польским логиком Яном Лукасевичем в 1920-х годах. Название "польская запись" связано с национальностью автора.

---

#### Основные характеристики

| Характеристика | Описание |
|----------------|----------|
| **Порядок операндов** | Совпадает с инфиксной записью |
| **Порядок операторов** | Определяется приоритетом и ассоциативностью |
| **Скобки** | Не требуются |
| **Вычислимость** | Простая стековая интерпретация |
| **Длина записи** | Обычно короче инфиксной (нет скобок) |

---

#### Сравнение форм записи

| Инфиксная | Префиксная (польская) | Постфиксная (обратная польская) |
|-----------|----------------------|--------------------------------|
| `a + b` | `+ a b` | `a b +` |
| `(a + b) * c` | `* + a b c` | `a b + c *` |
| `a + b * c` | `+ a * b c` | `a b c * +` |

---

#### Алгоритм вычисления ПФЗ (стековый)

```
Для каждого токена t в постфиксной записи:
    Если t — операнд:
        Поместить t в стек
    Если t — бинарный оператор:
        b = pop()  // правый операнд
        a = pop()  // левый операнд
        результат = a t b
        push(результат)
    Если t — унарный оператор:
        a = pop()
        результат = t a
        push(результат)
Конец цикла
Результат = pop()
```

**Пример вычисления** `a b c * +` (при a=2, b=3, c=4):

| Шаг | Токен | Стек | Действие |
|-----|-------|------|----------|
| 1 | a=2 | [2] | push |
| 2 | b=3 | [2,3] | push |
| 3 | c=4 | [2,3,4] | push |
| 4 | * | [2,12] | 3*4=12 |
| 5 | + | [14] | 2+12=14 |

**Результат:** 14

---

#### Преимущества ПФЗ

1. **Отсутствие скобок** — не нужно определять приоритет и ассоциативность при вычислении
2. **Простота интерпретации** — достаточно одного стека
3. **Высокая скорость** — линейный проход без возвратов
4. **Удобство для компиляции** — легко преобразуется в машинный код

---

### Вопрос 7. Каковы свойства постфиксной формы записи?

**Ответ:**

Постфиксная форма записи (ПФЗ) обладает следующими ключевыми свойствами:

---

#### Свойство 1: Отсутствие скобок

В ПФЗ скобки не используются, так как порядок выполнения операций полностью определяется позицией операторов относительно операндов.

```
Инфикс: (a + b) * (c - d)
ПФЗ:    a b + c d - *
```

---

#### Свойство 2: Однозначность

Любое выражение в ПФЗ имеет **единственный** способ вычисления (при условии известной арности операторов).

---

#### Свойство 3: Сохранение порядка операндов

Операнды в ПФЗ следуют в том же порядке, что и в исходном инфиксном выражении.

```
Инфикс: a / b + c * d
ПФЗ:    a b / c d * +
Операнды: a, b, c, d — порядок сохранён
```

---

#### Свойство 4: Определение порядка операций положением операторов

Чем раньше выполняется операция, тем **раньше** её оператор появляется в ПФЗ (при обходе слева направо).

```
Инфикс: a + b * c
Порядок: сначала *, потом +
ПФЗ:    a b c * +
```

---

#### Свойство 5: Легкость стековой интерпретации

ПФЗ может быть вычислена с использованием одного стека без необходимости хранения дерева разбора.

---

#### Свойство 6: Уникальность

Для каждого выражения существует **единственная** ПФЗ (если определены приоритеты и ассоциативность операций).

---

#### Свойство 7: Обратимость

По ПФЗ можно восстановить инфиксную запись (с добавлением скобок), но не единственным способом.

---

#### Свойство 8: Компактность

ПФЗ часто короче соответствующей инфиксной записи за счёт отсутствия скобок.

| Инфикс | Длина | ПФЗ | Длина |
|--------|-------|-----|-------|
| `(a+b)*(c+d)` | 11 | `a b + c d + *` | 11 |
| `((a+b)+c)+d` | 13 | `a b + c + d +` | 11 |

---

#### Свойство 9: Независимость от приоритетов при вычислении

При вычислении ПФЗ не требуется знать приоритеты операций — достаточно арности.

---

#### Таблица свойств

| Свойство | Инфиксная запись | Постфиксная запись |
|----------|------------------|---------------------|
| Нужны скобки | Да | Нет |
| Учёт приоритетов при парсинге | Да | Нет |
| Стек для вычисления | Сложный (2 стека) | Простой (1 стек) |
| Читаемость для человека | Высокая | Низкая |
| Читаемость для машины | Низкая | Высокая |

---

### Вопрос 8. В чем состоят отличия алгоритма преобразования управляющих конструкций в постфиксную запись от алгоритма преобразования арифметических выражений?

**Ответ:**

Алгоритмы преобразования **арифметических выражений** и **управляющих конструкций** (условных операторов, циклов, switch) в постфиксную запись имеют существенные отличия.

---

#### Основные отличия

| Характеристика | Арифметические выражения | Управляющие конструкции |
|----------------|--------------------------|-------------------------|
| **Тип операторов** | Бинарные, унарные | Условные переходы, метки |
| **Структура** | Линейная | Ветвящаяся, циклическая |
| **Генерация меток** | Не требуется | Обязательна |
| **Стек операторов** | Используется (алгоритм Дейкстры) | Не используется |
| **Обработка** | Слева направо | С разрывами последовательности |
| **Результат** | Один поток операций | Несколько блоков с переходами |

---

#### Отличие 1: Генерация меток

**Арифметические выражения:** метки не требуются.

**Управляющие конструкции:** необходимо генерировать уникальные метки для обозначения точек переходов.

```javascript
// Для условного оператора
var elseLabel = newLabel();  // метка для else-ветки
var endLabel = newLabel();   // метка для конца
```

---

#### Отличие 2: Операции переходов

**Арифметические выражения:** только операторы вычислений (+, -, *, /, и т.д.).

**Управляющие конструкции:** добавляются специальные операции передачи управления:

| Операция | Назначение |
|----------|-------------|
| `JMP` | Безусловный переход на метку |
| `JMPF` / `JMPT` | Условный переход по ложному/истинному условию |
| `CALL` | Вызов подпрограммы |
| `RET` | Возврат из подпрограммы |

---

#### Отличие 3: Структура выходного кода

**Арифметические выражения:** линейная последовательность.

```
a b c * + =
```

**Условный оператор:** нелинейная структура с метками.

```
cond elseLabel JMPF
thenBlock
endLabel JMP
elseLabel:
elseBlock
endLabel:
```

---

#### Отличие 4: Использование стека операторов

**Арифметические выражения:** активно используется стек операторов (алгоритм сортировочной станции) для учёта приоритетов.

**Управляющие конструкции:** стек операторов не используется; вместо этого используется **стек меток** для обработки вложенности.

```javascript
// Стек меток для вложенных конструкций
var labelStack = [];

function pushLabel(l) { labelStack.push(l); }
function popLabel() { return labelStack.pop(); }
function topLabel() { return labelStack[labelStack.length - 1]; }
```

---

#### Отличие 5: Учёт приоритетов

**Арифметические выражения:** критически важен приоритет операций (* выше, чем +).

**Управляющие конструкции:** приоритеты не применяются — важна только вложенность.

---

#### Отличие 6: Обработка пустых веток

**Арифметические выражения:** не имеют "пустых" частей.

**Условный оператор:** может иметь пустую else-ветку, что требует специальной обработки.

---

#### Отличие 7: Циклические конструкции

**Арифметические выражения:** не содержат циклов.

**Циклы:** требуют генерации:
- Метки начала цикла
- Метки выхода из цикла
- Операций инкремента счётчика
- Проверки условия с переходом

```
startLabel:
    cond endLabel JMPF
    body
    startLabel JMP
endLabel:
```

---

#### Отличие 8: Таблицы переходов (для switch)

**Арифметические выражения:** не требуют.

**Switch (множественный выбор):** может требовать генерации таблицы переходов или цепочки сравнений.

---

#### Пример сравнения

**Арифметическое выражение:**
```
Вход:  a + b * c
ПФЗ:   a b c * +
```

**Условный оператор:**
```
Вход:  if x > 0 then y = 1 else y = -1
ПФЗ:   x 0 > L1 JMPF y 1 = L2 JMP L1: y -1 = L2:
```

**Цикл:**
